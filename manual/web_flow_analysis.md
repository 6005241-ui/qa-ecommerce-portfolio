## 6. Синтез: что происходит, когда покупатель нажимает кнопку «Купить»?
На основе наблюдений за сетевым трафиком Wildberries:

Начало процесса — после выбора товара и нажатия кнопки «Купить» на карточке товара отправляется POST-запрос на /__internal/ru-basket-api/webapi/lk/basket/data_v2 с полными данными о товаре (артикул, характеристика, цена, скидка, количество). Сервер возвращает актуальное состояние корзины, подтверждая добавление.

Переход в корзину — если пользователь переходит в корзину, браузер отправляет POST-запрос на /__internal/ru-basket-api/lk/basket/spa/refresh_v2 с телом, содержащим те же данные о товаре (количество может быть изменено). В ответе приходит детальная структура корзины, включая итоговую сумму, доступные способы оплаты, условия доставки.

Начало оформления — при переходе на страницу оформления заказа (или при нажатии кнопки «Оформить» в корзине) отправляется POST-запрос на bnpl-gateway.wildberries.ru/bnpl/v1/offers-partial для получения предложений по рассрочке/кредиту. Этот запрос содержит информацию о корзине (сумму, список товаров) и возвращает доступные финансовые предложения.

Дальнейшие шаги — полное оформление заказа (выбор адреса, способа оплаты, финальное подтверждение) требует дополнительных запросов, которые в данном исследовании не были зафиксированы из-за прерывания процесса. Однако видно, что Wildberries использует микросервисную архитектуру: отдельные эндпоинты для корзины, для получения финансовых предложений, для расчёта доставки.

Ключевое наблюдение: все операции, кроме первоначальной загрузки страниц, проходят через POST-запросы с JSON-телами, что типично для современного SPA-приложения. Основной функционал (добавление в корзину, обновление количества, начало оформления) работает стабильно, несмотря на фоновые ошибки, связанные с перегрузкой API тегов (429) и отсутствием некоторых статичных файлов (404). Эти ошибки не влияют на критический путь покупки, но могут сигнализировать о проблемах с инфраструктурой или настройками кэширования.
